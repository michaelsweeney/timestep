# BND system visualizer (trial)

Standalone exploration of building a clickable HVAC system visualizer from the
EnergyPlus `.bnd` (Branch Node Details) report. Not wired into the app — this
directory is a self-contained prototype to evaluate whether the idea holds up.

## Run it

Open `index.html` in a browser (works from `file://`, needs network for the
cytoscape CDN scripts) and drop any `eplusout.bnd` onto the page. Sample files:
`test-models/large-office-multifreq/eplusout.bnd` and
`test-models/large-office-design-day/eplusout.bnd`.

- Pick a pre-seeded sample from the dropdown (generated by
  `node collect-samples.mjs` into a gitignored `samples.js`; `#sample=N` in
  the URL auto-loads one) or drop a file.
- Two layouts. **system flow** (default) is a deterministic rule-based
  schematic — same .bnd, same drawing: columns left→right are plant supply →
  AHU/airside → distribution (terminals, zone equipment) → zones → return
  path; one horizontal band per system (plant/condenser loops first, then air
  loops); topological rank orders each band; orthogonal (taxi) polyline
  edges; edges colored by fluid (blue water, green air, dashed red implicit
  loop interfaces). Components are classified by branch membership first,
  then node fluid type — so a VAV reheat coil draws in the air band with its
  water edge crossing columns, while an air-cooled chiller stays waterside.
  **organic** is the fcose force layout with compound loop boxes.
- Click a component: side panel shows its inlet/outlet nodes (with fluid type
  and suspicious-node flags) plus **every object touching those nodes** from
  the Parent / Non-Parent Node Connection records — setpoint managers,
  controllers, equipment connections. This is the "link it back to other
  objects" drill-down.
- Click an edge: the fluid node itself, and everything connected to it.
- Loop dropdown filters to one loop plus whatever is wired to it; dashed red
  edges are implicit interfaces (plant supply→demand crossover, air loop
  supply/return ↔ zone equipment node mappings).

## Is the .bnd still maintained? (researched 2026-06-11)

**Yes — no deprecation signals anywhere.** The half-remembered "not actively
maintained" claim is about the *Fortran HVAC-Diagram post-processor*, not the
file. Evidence:

- The writer code (`BranchNodeConnections.cc`, `ReportLoopConnections` in
  `SimulationManager.cc`) has substantive commits through **June 2026**:
  cycle-handling and sort-order fixes (PR#11066, merged 2025-07), new
  components added to the connection arrays (2025), the Node Namespace
  refactor (PR#11462, merged 2026-03), `std::format` migration (2026-06).
  Developers are refactoring *into* this code, not abandoning it.
- The record format is frozen-stable: the 25.2.0 `! <Record Type>` header set
  is **identical** (59 types, zero diff) to our 22.1-generated fixtures. The
  doc source hasn't needed an edit since 2016 because the format doesn't move.
- The 25.2 Output Details documentation still states the file is "intended to
  support software which draws a schematic diagram of the HVAC system" — this
  visualizer is the documented purpose.
- The official HVAC-Diagram tool still ships and works in 25.2, but it *is*
  legacy: its Python replacement (PR#9217) has been a stalled draft since
  2021. A third-party .bnd visualizer is more valuable for that, not less.
- Modern alternative to watch: the **HVACTopology** tabular report (24.2+,
  PR#10372) lands in `eplusout.sql` (`TabularDataWithStrings`) with air loop /
  zone equipment / plant loop component-arrangement tables. Coarser than the
  .bnd (no node numbers, fluid types, non-parent connections, suspicious
  nodes) but machine-queryable and the strategic direction — keep the parser
  behind a seam so a SQL-backed topology source can be swapped in for ≥24.2
  models.

## How the graph is built

`parsebnd.js` parses the self-documenting record types (every record has a
`! <Type>,<field>,...` header; data rows are trim+split-on-comma — IDF names
cannot contain commas). `buildgraph.js` turns the model into a graph:

- **Components are vertices, EnergyPlus nodes are edges.** In E+ a fluid node
  *is* the connection point, so an edge is drawn wherever one component's
  outlet node is another's inlet node.
- Vertices come from Component Sets, connector lists (splitters/mixers),
  supply/return air path components, controlled zones, and zone equipment
  lists. Components with no node rows anywhere in the structured records
  (return plenums) are backfilled from their own Inlet/Outlet node
  connections.
- Containers (components that parent other component sets) become compound
  nodes. Edge suppression is **per endpoint**: a container only registers
  endpoints its children don't cover, so an air terminal whose only child is
  its reheat coil keeps its uncovered inlet without duplicating the coil's
  outlet edge.
- Two kinds of implicit links join different node names: plant/condenser
  supply↔demand connections, and the AirLoop Supply/Return Connections that
  map air-loop-side nodes to zone-equipment-side nodes.

Validated on both large-office fixtures: 231 graph nodes, 260 edges, zero
orphaned flow vertices, full loop closure (fan → splitter → terminal → zone →
plenum → mixer → back to the air loop).

## Where this could go

- Click a component → pull its timeseries from the sibling `.sql` (the app
  already has the query layer; the cache dir even keeps the `.bnd` next to
  converted ESO files).
- Color components/edges by live data (surface of the same ask as ISSUE#20's
  color-surfaces-by-data).
- Graduate `parsebnd.js` into `packages/core` with tests, behind an engine
  seam like `readbnd.ts` (which today only extracts node→fluid-type).
